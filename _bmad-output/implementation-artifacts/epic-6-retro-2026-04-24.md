## What Went Well

- **Spec guardrails were exceptional.** Critical Guardrails section documented every non-obvious pitfall upfront: `z.uuid()`, no `withAdmin` on public endpoint, atomic SQL with `sql` template tag, `db:push` over `db:generate`. Zero build failures, zero DB mistakes on first pass.
- **Scope was right-sized.** One feature, one column, one server function, clear ACs. No scope creep.
- **Build verification was quantified.** 1,613 KiB confirmed in the story completion notes — not assumed, measured.
- **Parallel code review sub-agents delivered real value.** Three independent reviewers (Blind Hunter, Edge Case Hunter, Acceptance Auditor) running simultaneously caught issues a single-pass review would have missed.
- **Session hooks worked.** Context carried cleanly across sessions via save/resume hooks. `new-server-fn` skill scaffolded the server function correctly on first pass.

---

## Challenges

- **Spec had blind spots that CR caught.** Four items needed fixing post-implementation:
  1. Unhandled promise rejection — spec said "no `.catch()`" but didn't distinguish silencing from ignoring a rejection
  2. Draft posts incrementing count — spec didn't exclude admin previews
  3. No sessionStorage deduplication — refresh inflation not addressed
  4. Missing `useRef` Strict Mode guard — React 18 double-fire not covered

- **Sprint status not auto-updated after CR approval.** Required manual update at retro start.

- **Partial `z.uuid()` cleanup.** `admin.ts` and new schemas updated, but pre-existing `z.string().uuid()` in other `post.ts` schemas left inconsistent.

---

## Key Insights

1. **The review layer should not be the primary safety net.** Issues caught in CR (unhandled rejections, draft inflation, dedup gaps) should be anticipated in the spec's guardrails or during dev story creation. The spec author and dev agent have a shared blind spot: they implement what's written, not what's implied.

2. **Constant improvement mindset produces compounding gains.** sessionStorage dedup, useRef guard, draft-only check, and `.catch()` were all added during the CR pass — not required by the original spec but obvious improvements once surfaced. Asking "what edge cases did we miss?" at every review is worth the time.

3. **The parallel sub-agent review architecture is a force multiplier.** Three orthogonal lenses (adversarial, edge-case, acceptance) on the same diff catch more than one comprehensive reviewer. This pattern should be default for all stories, not just epic-closing ones.

4. **First-ever project retrospective.** Blog app v1 (21 stories, 6 epics) is complete. The BMAD workflow delivered a full production-ready application with consistent quality, bilingual support, admin tooling, SEO, and analytics from a well-structured plan.
